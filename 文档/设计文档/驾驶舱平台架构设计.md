# 驾驶舱平台架构设计

## 背景与目标

Cursor IDE 对非代码人员极不友好，但 Cursor CLI 操控电脑的能力极其强大。**驾驶舱平台**的核心思路是：Cursor CLI 做引擎，我们做驾驶舱——非技术人员不需要懂 Cursor，只需要打开网页对话。

本项目从现有的飞书桥接服务（龙虾军团）进化而来，从"飞书 → Cursor"单点桥接升级为"本地 Agent + 中心服务器 + 网页 UI"的完整平台。基于开源项目 claudecodeui 进行改造。

## 整体架构

```
┌──────────────────────────────────────────────┐
│                 中心服务器                      │
│                                              │
│  ┌─────────┐  ┌──────────┐  ┌────────────┐  │
│  │ Web UI  │  │ 消息路由  │  │ 用户/权限   │  │
│  │ (React) │  │(WebSocket)│  │  管理后台   │  │
│  └────┬────┘  └────┬─────┘  └────────────┘  │
│       │            │                         │
│       └──────┬─────┘                         │
└──────────────┼───────────────────────────────┘
               │ WebSocket (wss://)
    ┌──────────┼──────────────┐
    │          │              │
    ▼          ▼              ▼
┌────────┐ ┌────────┐  ┌────────┐
│员工A机器│ │员工B机器│  │员工C机器│
│        │ │        │  │        │
│本地Agent│ │本地Agent│  │本地Agent│
│   ↓    │ │   ↓    │  │   ↓    │
│Cursor  │ │Cursor  │  │Cursor  │
│  CLI   │ │  CLI   │  │  CLI   │
└────────┘ └────────┘  └────────┘
```

三层分离：Web UI 面向用户、中心服务器负责路由调度、本地 Agent 驱动 Cursor CLI 执行任务。

## 模块一：本地 Agent（最关键）

本质上是现有 relay-bot 的精简通用版，换掉飞书接口，改成 WebSocket 客户端。

### 职责

- 开机自启动（macOS launchd / Windows Service）
- 连接中心服务器（WebSocket + 自动重连）
- 接收任务 → 调用 Cursor CLI → 流式回传结果
- 定期上报心跳（在线状态、系统资源）

### 连接状态机

```
启动 → 连接服务器 → 已连接（正常工作）
          ↑            │
          │        断线检测
          │            ↓
      重连(退避)  ← 断线状态
    1s→2s→4s→...→60s
```

### 关键设计：任务执行与网络连接解耦

- Cursor CLI 在本地运行，不依赖网络
- 任务执行中断线 → 任务继续执行 → 结果本地缓存 → 重连后补发
- 用户体验：结果可能晚到，但不会丢

### 安装体验目标

```
员工下载 install.sh（或 install.exe）
→ 自动安装 Cursor CLI（如果没有）
→ 自动安装本地 Agent
→ 弹出浏览器让员工登录绑定
→ 完成，后台运行
```

## 模块二：中心服务器

### 技术选型

| 层 | 技术 | 选择原因 |
|---|------|---------|
| 运行时 | Bun | 团队已熟悉，TypeScript 直接运行 |
| 框架 | Hono 或 Elysia | Bun 原生，性能好 |
| WebSocket | Bun 内置 | 无需额外依赖 |
| 数据库 | SQLite → PostgreSQL | SQLite 起步，规模化后迁移 |
| 前端 | React（继承 claudecodeui） | 参考项目已有完整前端 |
| 部署 | 阿里云服务器 | 已有资源 |

### 核心功能

| 功能 | 说明 |
|------|------|
| 用户管理 | 注册、登录；**Cursor Key 由平台 Key 池调度**，不对终端用户暴露多 Key 细节 |
| 设备管理 | 本地 Agent 注册、在线状态、最后心跳 |
| 消息路由 | 用户消息 → 找到对应 Agent → 转发 |
| 会话管理 | 历史记录存储（Agent 执行结果存到服务器） |
| 权限控制 | 按用户配置能力范围（文件操作、Shell 等） |
| 用量统计 | 每人每天使用次数、token 消耗 |

## 关键决策

| 决策点 | 选择 | 备选方案 | 选择原因 |
|--------|------|---------|---------|
| 参考代码基础 | claudecodeui | 从零开发 | 已有完整 React 前端 + Cursor CLI 集成，大幅减少工作量 |
| 本地 Agent 运行时 | Bun | Node.js | 团队已熟悉，TypeScript 直接运行，启动快 |
| 通信协议 | WebSocket (wss://) | HTTP 轮询 / gRPC | 实时双向，适合流式输出，实现简单 |
| 初始数据库 | SQLite | PostgreSQL | 起步阶段够用，迁移成本低 |
| 首个支持平台 | macOS | 跨平台 | 团队主力设备，已有 launchd 经验，验证后再扩展 |
| Cursor CLI 授权 | **多 Cursor 账号 + 多 API Key + 欠费自动轮询** | 第三方 Key（已确认不可行）/ 单 Key | 官方 Key 池轮换，用户侧无忧可用 |

## 难点分析

### 难点一：Cursor CLI 授权模型（已拍板）

**结论（2026-03-18，农总确认）**：Cursor CLI **不支持**用第三方 API Key 替代 Cursor 官方 User API Key；该路径不再作为方案考虑。

**已定策略：多账号 Key 池 + 欠费自动轮询**

| 要点 | 说明 |
|------|------|
| 账号与 Key | 维护多个 Cursor 账号，每个账号在 Dashboard 生成 User API Key，形成 Key 池 |
| 轮询机制 | 调用 CLI 时绑定当前 Key；若返回欠费/额度类错误，自动切换池中下一 Key 重试 |
| 目标 | 终端用户无感，不因单账号欠费而整体不可用 |
| 实现落点 | 龙虾军团已有单 Key 欠费降级逻辑，驾驶舱/中心侧需扩展为**多 Key 池与健康度管理**（配置、轮换、审计勿写日志明文） |

| 方案（历史） | 状态 |
|-------------|------|
| 第三方 Anthropic/OpenAI Key 直连 CLI | ❌ 不可行，已排除 |
| 单 Cursor Key | 仅适合极小流量；规模化必须用池 |
| 每人独立 Cursor 账号 | 可选补充（大客户），与 Key 池可并存 |

### 难点二：跨平台部署

| 系统 | 自启动方案 | 安装复杂度 | Cursor CLI 支持 |
|------|-----------|-----------|----------------|
| macOS | launchd | 低（已验证） | 完整支持 |
| Windows | NSSM / Windows Service | 中等 | 支持 |
| Linux | systemd | 低 | 支持 |

策略：先只支持 macOS，验证整个流程跑通后再做 Windows。

### 难点三：安全

| 安全维度 | 措施 |
|---------|------|
| 传输加密 | WebSocket 必须用 `wss://`（TLS） |
| 身份认证 | Agent 连接时用 token 认证，token 有过期时间 |
| 指令过滤 | 服务器端过滤危险指令（`rm -rf /`、`sudo` 等） |
| 权限沙箱 | 限制 Cursor CLI 可操作的目录范围 |
| 审计日志 | 所有操作记录在案 |

## 风险与注意事项

- ~~Cursor CLI 授权模型~~ → 已明确为官方多 Key 池 + 轮询；工程重点是池管理与合规存储
- claudecodeui 是 GPL-3.0 协议，商用需评估合规性
- 跨平台部署会显著增加复杂度，应严格按阶段推进
- 安全设计不能后补，必须从 MVP 阶段就纳入
