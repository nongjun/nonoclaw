### 关于命名

虾群不贴切，叫龙虾军团吧，是 nonomax 公司旗下的 nonoclaw（龙虾军团）。

### 关于 templates 同步策略

templates 同步状况不能用文件数量来衡量，其实文件数量少是没问题的，因为模版里只需要保留核心的几个闭环，而不需要为编程而生，我们 root 维护虾 是为了编程和代码维护而生的。但我们的仓库 https://github.com/nongjun/feishu-cursor-claw 也已经为了记忆系统升级做了一些迭代。而我们维护虾为了精简做了一些迭代。因此我们不是简单同步，而是将我们最新的迭代符合用于做模版的，进行更新。

### 关于产品架构（完整表述）

我们这个项目是大娃套小娃，大圈套小圈的一个服务器和项目。首先目前最外圈也就是 root 目录下的 agent.md 和 rules、skills、command 以及记忆体系、心跳体系等系统功能、文档等都是为了维护整个服务器而存在的。

然后是在我们服务器上将搭建多个工作区，并对工作区进行分组，就是里面的小圈。每一个工作区都要在初始化时复制一遍 /root/templates 文件夹，让这个工作区独立能运转，独立有自己的记忆系统、心跳系统、灵魂、人设等等。

那么我们需要一个网页来注册和创建工作区，每个用户是一个分组，分组中可设置多个工作区，每一个工作区都是该用户可以管理的。而用户与用户之间需要隔离开。每个用户通过一个 claudecodeui 来管理 cursor cli。每一个 agent 我们都以"虾"字结尾来命名，"财务虾"、运营虾、分析虾等等，比如"维护虾"就是我们这个服务器 root 目录下的 agent.md、rules、记忆、飞书连接、skill、command 等等相关内容的一个集合体。

未来每个用户可以在我们服务器上在自己的名下建立多个工作区，每个工作区就是一只虾，每个用户还会有一只虾总管，负责创建工作区、编排各工作区即每只虾的早期记忆和分工。

用户可以选择每一只虾是在云端的还是在他本地电脑的，在本地电脑的则文件不会上传到我们服务器，我们帮他一键部署 cursor cli、记忆体系、飞书连接甚至以后有钉钉连接、企微连接等插件。但都通过我们服务器的网页来方便管理，内核都是 claudecodeui，到时候我们进行详细的改装。

## 第二部分：架构理解与落地指导

基于农总的设计意图，以下是技术团队的架构理解，作为落地实施的指导。

### 2.1 产品定位

**龙虾军团（nonoclaw）** 是 nonomax 公司的多租户 AI Agent 平台，让每个用户拥有自己的 AI 助手军团。

核心理念：**大圈套小圈** — 一个服务器托管多个用户，每个用户管理多个独立 AI 工作区（"虾"），每只虾有独立的记忆、人格、能力。

### 2.2 三层架构

```
┌─────────────────────────────────────────────────┐
│ 第一层：服务器层（/root）— 维护虾                  │
│ 职责：服务器维护、平台运营、用户管理                 │
│ 文件：AGENTS.md、rules、skills、commands          │
│       server.ts、飞书连接、记忆/心跳/调度体系       │
│ 特点：为编程和代码维护而生，包含全部开发工具链        │
├─────────────────────────────────────────────────┤
│ 第二层：用户层 — 虾总管                            │
│ 职责：创建工作区、编排分工、管理用户的所有虾          │
│ 入口：Web 管理面板（claudecodeui 改装）             │
│ 特点：每个用户一个虾总管，用户之间完全隔离            │
├─────────────────────────────────────────────────┤
│ 第三层：工作区层 — 各种虾（财务虾、运营虾、分析虾…）  │
│ 职责：执行具体业务任务                              │
│ 初始化：从 /root/templates 复制核心闭环              │
│ 特点：独立记忆 / 独立人格 / 独立心跳 / 独立调度       │
│ 部署：云端（在本服务器）或 本地（用户电脑）           │
└─────────────────────────────────────────────────┘
```

### 2.3 templates 设计原则

templates 是每只虾的"出厂 DNA"，遵循以下原则：

| 原则 | 说明 |
|------|------|
| **只含核心闭环** | 记忆、心跳、调度、会话协议、身份系统、工具能力 |
| **不为编程而生** | 不包含语言专属规则、开发 agents/commands/skills |
| **轻量** | 文件数量少没问题，核心闭环完整就行 |
| **选择性同步** | 不是简单复制 root → templates，而是精选适合模板的迭代更新 |
| **独立运转** | 复制到新工作区后即可独立工作，有自己的记忆/心跳/灵魂/人设 |

### 2.5 虾的命名规范

所有 Agent 以"虾"字结尾命名：

| 角色 | 命名 | 部署位置 | 职责 |
|------|------|---------|------|
| 服务器维护 | 维护虾 | 服务器 /root | 代码维护、平台运营 |
| 用户管理 | 虾总管 | 服务器 /用户目录 | 创建/编排/管理该用户的所有虾 |
| 业务执行 | 财务虾、运营虾、分析虾… | 云端或本地 | 执行具体业务任务 |

### 2.6 部署模式

| 模式 | 说明 | 文件存储 |
|------|------|---------|
| **云端虾** | 工作区在 nonomax 服务器上 | 文件在服务器 |
| **本地虾** | 工作区在用户本地电脑 | 文件留在本地，不上传 |

两种模式都通过 nonomax 服务器的 Web 面板管理。本地虾通过一键部署安装 Cursor CLI、记忆体系、IM 连接插件（飞书、钉钉、企微等）。

### 2.7 Web 管理面板

基于 claudecodeui 改装，功能包括：

- 用户注册/登录
- 工作区（虾）的创建/删除/配置
- 每只虾的状态监控
- IM 插件管理（飞书/钉钉/企微连接）
- 本地虾的一键部署
- 用户间完全隔离


### 2.9 隔离策略：目录级 + 能力限制

不使用 Docker 容器（轻量服务器容量有限），采用"能力阉割 + 目录隔离"方案：

**云端虾（受限模式）：**
- 工作区：`/home/用户名/workspaces/虾名/`
- Cursor CLI 禁用 Shell 工具（通过 `workspace-rules.mdc` 强制约束）
- 允许：文件读写（限定在自己目录内）、WebSearch、WebFetch
- 禁止：Shell 命令、SSH、任意进程执行
- 记忆系统：server.ts 统一索引，虾通过内部 API 查询（不需要自己跑 bun memory-tool.ts）
- 目录隔离：每用户一个 Linux 用户，目录权限隔离

**本地虾（完整模式）：**
- 全功能：Shell、网络、文件、一切
- 文件留在用户电脑，云端只存管理元数据

**容量估算：** 一台 2C4G 轻量服务器可支撑 20-30 只云端虾同时运行。

### 2.10 API Key 轮询池

平台维护多个 Cursor API Key，按请求轮询分配：
- 用户付费后获得 token 额度
- 每次 Agent 调用消耗 token
- Key 池支持热添加/移除
- 单 Key 欠费/限流时自动切到下一个

### 2.11 虾总管架构

虾总管是每个用户的"指挥官"，本身也是一个 Cursor CLI 工作区：

```
/home/用户名/
|-- manager/                    # 虾总管工作区
|   |-- .cursor/MEMORY.md       # 虾总管自己的记忆
|   |-- .cursor/SOUL.md         # 虾总管人格
|   |-- roster.json             # 虾名册：所有虾的路径、角色、状态
|   +-- AGENTS.md               # 虾总管职责说明
|
+-- workspaces/
    |-- 财务虾/
    |   |-- .cursor/
    |   |   |-- ORDERS.md       # <-- 虾总管写入的指令
    |   |   |-- REPORT.md       # <-- 虾向虾总管汇报
    |   |   +-- MEMORY.md       # 虾自己的记忆
    |   +-- cron-jobs.json
    +-- 运营虾/
        +-- ...
```

**虾总管与虾的通信（文件即通信）：**

| 方向 | 方式 | 实现 |
|------|------|------|
| 虾总管 -> 虾 | 写入 `.cursor/ORDERS.md` | 虾在会话启动时读取并执行 |
| 虾总管 -> 虾（紧急）| 写入 `cron-jobs.json` | 调度器自动触发虾执行 |
| 虾 -> 虾总管 | 写入 `.cursor/REPORT.md` | 虾总管心跳巡检时读取 |
| 虾总管巡检 | 读取各虾 MEMORY.md | 了解各虾近况 |

**权限模型：** 虾总管对该用户下所有虾目录有读写权限，虾之间互相无权限（上级看下级，平级隔离）。

### 2.12 claudecodeui 改装策略

基于 `/root/参考代码/claudecodeui`（已汉化版），最小化改动：

- 保留现有技术栈（React + Node.js）
- 复用现有多工作区管理能力（每个工作区 = 一只虾）
- 复用现有 Cursor CLI 连接逻辑
- 复用现有文件管理 UI
- 新增：用户注册/登录（手机号 + 微信 OAuth）
- 新增：付费网关集成
- 新增：虾总管管理面板
- 新增：IM 插件配置 UI

### 2.14 IM Adapter 接口

抽象通用 IM 适配器接口，每只虾可独立配置：

```
interface IMAdapter {
  connect(config: IMConfig): Promise<void>
  onMessage(handler: MessageHandler): void
  sendCard(card: CardPayload): Promise<void>
  sendText(text: string): Promise<void>
  downloadMedia(mediaKey: string): Promise<Buffer>
}
```

当前实现：飞书（WebSocket 长连接模式）
计划支持：钉钉、企微

### 2.8 后续开发优先级（建议）

1. **templates 同步机制** — 建立 templates 更新流程，每次核心闭环迭代后精选同步
2. **工作区初始化脚本** — 从 templates 创建新工作区的自动化脚本
3. **多租户隔离** — 用户目录结构、权限隔离、Cursor CLI 实例管理
4. **虾总管逻辑** — 用户级 Agent，管理该用户名下的所有工作区
5. **claudecodeui 改装** — 多用户 Web 面板
6. **IM 插件体系** — 飞书之外，增加钉钉、企微连接
