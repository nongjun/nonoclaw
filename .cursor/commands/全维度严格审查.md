# 全维度严格审查

对代码进行 6 个维度的严格审查，每个维度启动一个独立子 agent 并行执行。

## 执行策略：6 个子 agent 并行

同时启动 6 个子 agent，各自负责一个维度，互不干扰，最后汇总。

| Agent | 审查维度 | 关注点 |
|-------|---------|--------|
| Agent 1 | 代码健康 | 死代码/冗余/废弃/空实现/过时依赖/硬编码/命名混乱 |
| Agent 2 | 功能可用 | 流程完整性/空状态/加载反馈/错误提示/边界处理/移动适配 |
| Agent 3 | 自动化 | 手动步骤可否自动化/定时任务/批量能力/合理默认值/自动刷新 |
| Agent 4 | 交互体验 | 操作确认/即时反馈/撤销能力/搜索筛选/快捷操作 |
| Agent 5 | 错误体验 | 用户友好提示（禁止裸露HTTP状态码）/分场景引导（权限不足→联系管理员、登录过期→重新登录、服务异常→稍后重试）/全局兜底页 |
| Agent 6 | 企业价值 | 数据准确/权限管控/审计追溯/数据导出/业务连续性/合规安全 |

## 严重程度分级

每个问题标注严重程度：
- 🔴 **阻断** — 必须修复才能发布（安全漏洞、数据错误、流程中断）
- 🟡 **警告** — 强烈建议修复（体验缺陷、缺少兜底、硬编码）
- 🔵 **建议** — 锦上添花（优化空间、更好的默认值）

## 输出格式

每个子 agent 输出各自维度的问题清单后，汇总为一份总报告：

```
# 全维度严格审查报告

审查范围：[文件/模块]
时间：YYYY-MM-DD HH:MM

## 汇总

| 维度 | 🔴 阻断 | 🟡 警告 | 🔵 建议 | 总计 |
|------|---------|---------|---------|------|

| 功能可用 | X | X | X | X |
| 自动化 | X | X | X | X |
| 交互体验 | X | X | X | X |

| 企业价值 | X | X | X | X |

（Agent 1 的详细发现）

### 2. 功能可用

...

## 修复优先级建议
```

## 原则

- 先策略后执行——报告输出后，等用户确认哪些要改再动手
- 不吹毛求疵——代码风格交给 formatter，审查关注实质问题
- 每个问题必须附带具体位置和修复方向
- 按优先级排序，让用户知道先改什么
