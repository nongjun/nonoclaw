# 规则沉淀

对刚才的问题解决过程进行复盘，提炼可复用的规则，录入对应的规则文档。

## 触发条件

- 刚解决了一个耗时较长或反复试错的问题
- 踩了一个"本可避免"的坑
- 发现了一个"如果当初知道就好了"的知识点

## 执行步骤

### 1. 复盘问题本质

- 根因是什么（不是"做错了什么"，而是"为什么会做错"）
- 时间花在哪（排查方向错误？知识盲区？）
- 什么情况下会遇到这个问题

### 2. 提炼可复用规则

原则：
- 一句话说清——如果需要解释，说明还没提炼到位
- 有明确触发点——在什么情况下应该想起这条规则
- 有具体动作——告诉未来的自己"做什么"而非"别做什么"

### 3. 自动识别归属

根据当前项目和问题类型，自动判断规则应写入哪个文件：

| 项目/场景 | 目标文件 |
|-----------|---------|
| 业务系统通用 | 项目文档目录/核心信念/团队基线.md |
| 小程序相关 | 对应模块/规则沉淀.md |
| Mac 应用相关 | 对应模块/规则沉淀.md |
| 其他项目 | 项目根目录/文档/规则沉淀.md |

如果目标文件不存在，自动创建。

### 4. 检查去重（必须先做！）

写入前搜索目标文件，检查是否已有类似规则：
- 已有完全相同的 → 不添加，告知用户
- 已有类似但不完整 → 补充现有规则
- 已有相关可合并 → 合并为子场景
- 确实没有 → 新增

### 5. 写入并确认

- 追加到目标文件对应分类下
- 标注日期
- 向用户展示：问题复盘 + 新增规则 + 写入位置

## 规则质量标准

- 不要太具体（某个变量名拼错）→ 要提炼出普适性
- 不要太抽象（要仔细）→ 要有可执行的动作
- 目标：让一个月后的自己或新同事直接受益
