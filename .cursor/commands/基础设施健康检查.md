# 基础设施健康检查

> 对 Docker 容器、数据库、反向代理、SSL 证书等基础设施进行全面健康检查

---

## 🚀 执行策略：多 Agent 并行

启动 **4 个子 Agent** 并行检查（`readonly: true`，`model: fast`）：

| Agent | 检查范围 | 优先级 |
|-------|---------|-------|
| Agent 1 | Docker 容器 + 数据库 + 反向代理 | P0 |
| Agent 2 | 业务服务 + 定时任务 | P1 |
| Agent 3 | 外部依赖 + SSL 证书 | P1/P2 |
| Agent 4 | 资源监控 + 常见问题诊断 | P2 |

---

## 🏗️ 自动发现系统拓扑

执行以下命令自动获取当前部署情况，无需硬编码：

- `docker ps --format '{{.Names}}\t{{.Status}}\t{{.Ports}}'` — 获取所有运行中的容器、状态与端口映射
- `docker compose config` 或 `cat docker-compose*.yml` — 获取完整服务定义
- `nginx -T 2>/dev/null` 或 `docker exec <nginx容器> nginx -T` — 获取 Nginx 配置（域名、上游、SSL 路径）
- 检查 `.env` 文件获取数据库连接信息

---

## 🔍 检查思路

### Agent 1：基础设施（P0）

**容器检查**：
- 所有 Docker 容器是否运行中（无 Restarting / Exited / unhealthy）
- 各后端服务健康端点是否返回 200（自动从容器端口发现 `/health` 或 `/api/health`）
- 容器日志是否有持续报错（最近 50 行内的 ERROR / FATAL）

**数据库检查**：
- MySQL / PostgreSQL 连接是否正常
- 连接数是否在合理范围内
- 是否有慢查询或锁等待

**反向代理检查**：
- Nginx 配置语法是否正确（`nginx -t`）
- 所有配置的上游（upstream）是否可达
- 错误日志中是否有 502 / 504 / "Connection refused"

### Agent 2：业务服务（P1）

**核心服务检查**：
- 各后端容器的环境变量是否完整加载（检查关键的 TOKEN / KEY / SECRET 变量）
- 服务日志中是否有"签名验证失败"、"解密失败"、"认证失败"等错误

**定时任务检查**：
- 是否有任务管理接口（如 `/api/jobs`），检查任务状态
- 是否有状态为 running 超过 30 分钟的卡死任务

### Agent 3：外部依赖（P1/P2）

**第三方服务检查**：
- 项目依赖的外部 API 是否可连通（从配置或环境变量发现 API 端点）
- API Key 是否已配置（非空）
- 最近日志中的外部调用是否有高失败率

**SSL 证书检查**：
- 所有 HTTPS 域名的证书剩余天数（< 7 天紧急，< 30 天警告）
- 证书自动续期工具（Certbot 等）是否正常运行

### Agent 4：资源与诊断（P2）

**资源监控**：
- 磁盘使用率（> 90% 紧急，> 80% 警告）
- 容器内存 / CPU 使用情况（`docker stats --no-stream`）
- 日志目录大小（是否需要清理）

**常见问题诊断**：
- 各服务间共享的密钥/配置是否一致（如 JWT_SECRET）
- 容器间通信是否使用了正确的内部端口（而非主机映射端口）
- DNS 解析是否正常

---

## ⚡ 模式参数

- `--quick`：仅检查 P0（容器 + 数据库 + 反向代理）
- `--fix`：检查后尝试自动修复常见问题（重启卡住容器、reload nginx、清理磁盘空间）
