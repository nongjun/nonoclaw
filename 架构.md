# nonoclaw — 系统架构

> 飞书 → Cursor AI 远程遥控桥接服务。用户通过飞书即时通讯发送消息（文字、语音、图片、文件），服务自动转发给本地 Cursor Agent CLI 执行，结果通过飞书卡片实时回传。

## 仓库四层架构

本仓库同时包含 nonoclaw 产品和开发工具链。四层文件架构详见 `AGENTS.md`「仓库架构：产品 vs 马鞍」，此处仅列要点：

- **产品**（`templates/`）= nonoclaw 本体，用户 clone 后获得的全部内容
- **源码**（`*.ts`, `feishu/` 等）= 运行代码，所有实例共享
- **马鞍工程**（`.cursor/`, `文档/`, `架构.md`, `AGENTS.md`, `README.md`）= 开发/迭代工具链
- **实例数据**（`.cursor/MEMORY.md`, `cron-jobs.json`, `.env` 等）= gitignored，每个部署私有

## 业务域划分

| 业务域 | 子系统 | 职责 | 说明 |
|-------|--------|------|------|
| **Gateway** | gateway.ts | 飞书 WebSocket 长连接、消息去重/解析、Worker 进程管理、飞书操作 HTTP API | 稳定层，极少重启；start.ts 为 launchd 入口 |
| **Worker** | server.ts | 指令处理、Agent 调度、记忆/定时/心跳/蒸馏、流式卡片 | 业务层，可随时重启；支持独立模式（向后兼容） |
| **API 桥接** | bridge.ts | OpenAI 兼容 API → Cursor Agent CLI → OpenAI 格式响应 | 供 OpenClaw 等外部系统以标准 API 方式调用 |
| **飞书集成** | feishu/ | 飞书 SDK 封装、消息收发、流式卡片、@提及、表情回应、策略管控 | 17 个文件，完整封装飞书开放平台交互细节 |
| **记忆系统** | memory.ts, memory-tool.ts, distill-chats.ts | 向量嵌入 + BM25 混合搜索、增量索引、嵌入缓存、对话蒸馏 | SQLite 存储，Cursor Agent 通过 CLI 工具调用；定时从 Cursor 对话记录提取记忆 |
| **自动化调度** | scheduler.ts | 定时任务执行（at / every / cron 三种调度模式） | 读取工作区 cron-jobs.json，自动触发 Cursor Agent |
| **心跳巡检** | heartbeat.ts | 定期触发 Cursor Agent 执行心跳检查清单 | 按协议决定是否通过飞书汇报异常 |
| **语音处理** | server.ts 内置 | 语音消息 → STT 转文字 → 作为文本指令执行 | 火山引擎豆包 STT 为主，本地 whisper-cpp 兜底 |
| **工作区模板** | templates/ | 新工作区初始化模板：人格、身份、记忆、规则、技能 | Cursor Agent 自动加载的完整配置体系 |
| **桌面操控** | plugins/turix-cua/ | AI 视觉理解 + 鼠标键盘操控，自然语言驱动 macOS 桌面自动化 | Python（Conda），截屏→AI→操控循环 |
| **运维部署** | service.sh, setup.sh | macOS launchd 服务管理、开机自启、崩溃恢复 | 纯 Shell 脚本，零外部依赖 |

## 模块依赖关系

### 主链路（Gateway-Worker 双进程）

1. **飞书 → gateway.ts**：飞书开放平台通过 WebSocket 长连接推送消息事件，`gateway.ts` 接收并完成消息去重、解析
2. **gateway.ts → server.ts（Worker）**：Gateway 通过 HTTP POST `/message` 将解析后的消息转发给 Worker 进程
3. **server.ts → 语音处理**：语音消息先发往火山引擎豆包 STT（WebSocket 二进制协议），失败时回退到本地 whisper-cpp
4. **server.ts → Cursor Agent CLI**：通过 `spawn` 调用本地 `~/.local/bin/agent`，使用 `--resume` 保持会话连续性
5. **server.ts → gateway.ts（飞书 API）**：Worker 通过 `GATEWAY_URL` 调用 Gateway 的 HTTP API（`/feishu/reply`、`/feishu/update`、`/feishu/send`、`/feishu/download`）操作飞书
6. **Worker 管理**：Gateway 负责 spawn Worker、健康检查、崩溃自动重启、重启期间消息队列缓冲

### API 桥接链路

- **外部系统 → bridge.ts → Cursor Agent CLI**：接收 OpenAI Chat Completions 格式的 HTTP 请求，转发给 Cursor Agent 执行，返回 OpenAI 格式响应

### 记忆系统（多层架构）

每次会话都是全新进程，**文件就是记忆**。记忆按从抽象到物理分为六层：

#### 层 0 — 协议层（驱动记忆行为）

| 文件 | 作用 |
|------|------|
| `.cursor/rules/memory-protocol.mdc` | 何时搜索、何时写入、长对话 flush 规则 |
| `AGENTS.md`「记忆闭环」| 会话启动读什么、蒸馏周期、经验沉淀路径 |

#### 层 1 — 身份与长期记忆（Markdown，实例私有，gitignore）

| 文件 | 写入者 | 读取时机 |
|------|--------|----------|
| `.cursor/SOUL.md` | 人类 / Agent | 每次会话启动 |
| `.cursor/USER.md` | Agent | 每次会话启动 |
| `.cursor/IDENTITY.md` | 首次出生仪式 | 每次会话启动 |
| `.cursor/MEMORY.md` | Agent 编辑 + 蒸馏追加 | 会话启动 + 向量检索 |

#### 层 2 — 日记与中期记忆（按日 Markdown）

| 文件 | 写入者 | 生命周期 |
|------|--------|----------|
| `.cursor/memory/YYYY-MM-DD.md` | `appendDailyLog()` + Agent | 按日累积，精华升到 MEMORY.md |
| `.cursor/memory/_chat-extract.md` | `distill-chats.ts` | 临时，蒸馏成功即删 |
| `.cursor/memory/distill-state.json` | 蒸馏脚本 | 持久，增量蒸馏游标 |

#### 层 3 — 会话留痕（Cursor CLI 自动记录）

`.cursor/projects/root/agent-transcripts/`：Cursor CLI 自动保存完整对话记录，每个会话一个 JSONL 文件。只读参考，**不进入向量索引**（噪音太高，有价值的部分通过蒸馏升到层 1）。

#### 层 4 — 可检索向量索引（SQLite）

`.memory.sqlite`：`chunks` 表（文本片段 + float32 向量 BLOB）+ `chunks_fts`（FTS5 全文检索）+ `embedding_cache`（嵌入缓存避免重复 API 调用）。

检索算法：0.7 × 余弦相似度 + 0.3 × BM25 关键词分，minScore 0.25~0.3。

索引范围：工作区非隐藏目录的文本文件 + 显式扫描 `.cursor/MEMORY.md` 和 `.cursor/memory/*.md` 日记。Agent 通过 `memory-tool.ts` CLI 的 `search/recent/write/index` 命令调用。

#### 层 5 — 外部对话源（只读）

`~/.cursor/chats/<workspace_hash>/<session>/store.db`：Cursor IDE 的对话存储。`distill-chats.ts` 从中提取 user/assistant 对话 → `_chat-extract.md` → Agent 蒸馏到 MEMORY.md。与层 3 的 sessions JSONL 是两套独立来源。

#### 数据流动

```
对话发生
  ├→ sessions JSONL（层3，原始日志）
  └→ Cursor IDE store.db（层5）
       ↓ 每12h distill-chats.ts 提取
       _chat-extract.md（层2，临时）
       ↓ Agent 蒸馏
       MEMORY.md（层1，精炼知识）
       ↓ scanMemoryPaths()
       .memory.sqlite（层4，向量可检索）
       ↓ memory-tool search
       Agent 检索召回
```

#### 模块调用链

- **Cursor Agent → memory-tool.ts → memory.ts → SQLite**：Agent 通过 shell 调用实现记忆的写入、搜索和索引
- **memory.ts → 火山引擎嵌入 API**：文本转向量，结果缓存在 SQLite 中避免重复计算
- **server.ts → distill-chats.ts → Cursor Agent CLI**：每 12 小时自动提取对话 → 蒸馏到 MEMORY.md

### 自动化系统

- **scheduler.ts → cron-jobs.json → Cursor Agent CLI → feishu/send.ts**：定时读取工作区的任务配置，到期时 spawn Cursor Agent 执行，结果通过飞书推送
- **heartbeat.ts → Cursor Agent CLI → HEARTBEAT.md → feishu/send.ts**：定期触发 Agent 执行心跳检查清单，Agent 按协议判断是否需要汇报

### 模块间无循环依赖

所有模块单向依赖。`gateway.ts`（通过 `start.ts`）和 `bridge.ts` 是两个独立入口；`server.ts` 由 Gateway spawn 或独立运行。

### 部署要点

- **Gateway-Worker 双进程**：`start.ts` 为 launchd 入口，import `gateway.ts`（稳定层）；Gateway spawn `server.ts`（业务层）。Worker 可随时重启，飞书连接不断
- **向后兼容**：`server.ts` 检测 `GATEWAY_URL` 环境变量——存在则 Worker 模式，不存在则独立模式（原有行为）
- **无公网依赖**：飞书通过 WebSocket 长连接主动推送，不需要公网 IP 或域名

| 差异项 | macOS 版 | 服务器版（Linux） |
